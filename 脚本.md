如果你的节点类型是固定的（如 Add, Subtract, Log），你可以为每个 NodeModel 写一个 `process()` 函数。
*   遍历图结构。
*   从 `Start` 节点开始，沿着 `Flow` 连接线寻找下一个节点。
*   调用下一个节点的 `process()`。
*   如果需要数据，回溯数据连接线获取 `Data` 节点的值。


执行流（Exec）的节点添加 portConnectionPolicy 重写，遵循以下原则：

输入端口 (Input):
如果是 Exec 类型：设置为 ConnectionPolicy::Many（允许多个上游触发同一个节点，逻辑上的 "Merge"）。
如果是 Data 类型：设置为 ConnectionPolicy::One（数据必须来源唯一）。

输出端口 (Output):
如果是 Exec 类型：设置为 ConnectionPolicy::One（强制单线执行，避免逻辑产生歧义的分叉）。
如果是 Data 类型：设置为 ConnectionPolicy::Many（变量可以被多处复用）。

在节点编辑器设计中，区分 **数据流 (Data Flow)** 和 **执行流 (Execution/Control Flow)** 的根本依据是：**副作用 (Side Effects)** 和 **时序依赖 (Temporal Dependency)**。

简单来说：
*   **执行流 (Execution Flow):** 决定 **“什么时候做”**。必须现在算
*   **数据流 (Data Flow):** 决定 **“用什么做”**。什么时候算都行

以下是具体的判断标准和设计原则：

---

### 1. 黄金判断标准：是否有“副作用”？

在 C++ 中，你可以通过函数的性质来判断：

#### 情况 A：纯函数 (Pure Function) -> **数据流**
如果一个函数满足以下条件：
1.  **没有副作用：** 调用它**不会**改变程序的状态（不修改成员变量、不写文件、不打印日志、不生成物体）。
2.  **确定性：** 只要输入参数相同，返回值永远相同。
3.  **只读：** 通常对应 C++ 中的 `const` 成员函数或静态工具函数。

*   **典型例子:** `a + b`, `sin(x)`, `Vector3.Distance`, `GetComponent`, `ToString`。
*   **节点表现:** 没有绿色的执行箭头，只有左边的数据输入点和右边的数据输出点。
*   **运行机制:** **“拉取模式 (Pull)”**。只有当后续节点需要这个数据时，这个节点才会计算。

#### 情况 B：有副作用 (Side Effects) -> **执行流**
如果一个函数做了以下事情：
1.  **改变状态：** 修改了变量的值 (`SetHealth`)。
2.  **影响世界：** 生成/销毁物体 (`Instantiate`, `Destroy`)，播放声音，施加力 (`AddForce`)。
3.  **控制流程：** `if`, `for`, `while`, `wait`。
4.  **入口事件：** `Start`, `Update`, `OnCollisionEnter`。

*   **典型例子:** `Debug.Log`, `transform.Translate`, `list.Add`, `variable = 5`。
*   **节点表现:** **必须**有左边的入口箭头（触发）和右边的出口箭头（完成）。
*   **运行机制:** **“推送模式 (Push)”**。上一个节点执行完，主动触发当前节点。

---

### 2. 实战分类表 (C++ 映射到节点)

为了方便你改代码，我们可以建立一个映射表：

| C++ 函数特征 | 归类 | 节点外观 (Socket) | 例子 |
| :--- | :--- | :--- | :--- |
| **Getter** <br> `int getHealth() const` | **数据流** | 左: 无 (或 `Target`) <br> 右: `Value` | 获取玩家血量 |
| **Setter** <br> `void setHealth(int v)` | **执行流** | 左: `Exec`, `Target`, `Value` <br> 右: `Exec` | 设置玩家血量 |
| **Math/Utils** <br> `float max(float a, float b)` | **数据流** | 左: `Val A`, `Val B` <br> 右: `Result` | 数学运算 |
| **Action** <br> `void Attack()` | **执行流** | 左: `Exec`, `Target` <br> 右: `Exec` | 播放攻击动画 |
| **Event** <br> `void mousePressEvent(...)` | **执行流 (起始)** | 左: 无 <br> 右: `Exec`, `EventData` | 鼠标点击事件 |
| **Branch** <br> `if (bool)` | **执行流 (控制)** | 左: `Exec`, `Condition` <br> 右: `True`, `False` | If / Switch |

---

### 3. 特殊情况：混合型节点

有些函数既返回数据，又改变状态。**这种必须归类为执行流**。

**例子：`List.Pop()` (弹出列表最后一个元素)**
*   它改变了列表（副作用 -> 执行流）。
*   它返回了弹出的元素（返回值 -> 数据流）。

**节点设计：**
*   **Input:** `Exec` (触发弹出动作), `List` (目标列表)。
*   **Output:** `Exec` (弹出完成), `Item` (弹出的那个数据)。

**为什么不能做成纯数据节点？**
如果 `Pop` 是数据节点，它是被动触发的。如果你把 `Pop` 的输出连到了两个不同的地方，后端可能会执行两次 `Pop`，导致列表被弹出了两次，这会导致严重的 Bug。**凡是改变状态的操作，必须通过执行流严格控制执行次数。**

---

### 总结

*   **只读、计算、无状态** -> **数据流** (圆点)。
*   **写入、动作、有时序** -> **执行流** (箭头)。
*   **既读又写** -> **执行流** (带数据的箭头节点)。

在设计你的 C++ 节点库时，问自己一个问题：**“如果这一帧我不连它的输出，它还需要运行吗？”**
*   如果不需要运行（比如 `a+b`，没人用结果就算了） -> **数据流**。
*   如果必须运行（比如 `PlaySound`，不连输出也要听到声音） -> **执行流**。
